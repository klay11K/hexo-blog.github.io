MySQL 面试题

MySQL 基础

MySQL 的优点？

  1. 免费开源。
  2. 社区活跃，生态完善。
  3. 支持事物。
  4. 支持分库分表，读写分离。

MySQL 的基础架构？

MySQL 的基础结构主要分为三层：

  1. 连接层，用于客户端与数据库建立连接。期间会校验主机名，用户名，密码是否正确，是否有权限访问等。
  2. 服务层，用于执行客户端发送过来的 SQL 语句。期间会对 SQL 进行语法解析和优化，并进行执行。
  3. 存储引擎层，真正与数据进行交互的地方。存储引擎层是可拔插的，一张表可以设置不同的存储引擎，存储引擎主要对文件系统进行存取数据。

一条 SQL 在 MySQL 中的执行流程是怎样的？

  1. 首先连接层会建立起客户端与数据库的连接。校验主机名，用户名和密码，权限等。
  2. 建立连接完毕后，进入到服务层，会先去查询缓存（MySQL 8.0 之前），如果这条 SQL 在缓存中命中了，则直接返回结果。
  3. 如果没有命中缓存，服务层会生成语法树对此 SQL 进行语法解析，解析失败就会抛出语法错误的信息。
  4. 解析成功，优化器会对此 SQL 进行优化，并生成一条执行计划。
  5. 生成执行计划后，会交由执行器来执行 SQL。
  6. 执行 SQL 后，存储引擎会对文件系统进行数据的存取，得到相应的结果。
  7. 得到结果后，会对缓存进行更新（MySQL 8.0 之前），再返回给客户端。

MyISAM 和 InnoDB 有什么区别？

  1. InnoDB 支持事物，MyISAM 不支持。
  2. InnoDB 支持行级锁，MyISAM 只支持表级锁。
  3. InnoDB 并发效率更高，MyISAM 并发效率低。
  4. InnoDB 支持数据库异常崩溃后的安全恢复，MyISAM 不支持。
  5. InnoDB 支持外键，MyISAM 不支持。

MyISAM 和 InnoDB 如何选择？

  1. InnoDB 适合业务复杂，需要事务支持的场景。
  2. MyISAM 适合以新增和查询为主的简单业务场景。

索引

什么是索引？索引的作用？

索引是一种用于快速查询和检索数据的数据结构，其本质可以看成是一种排序好的数据结构。

索引的作用就相当于书的目录。打个比方：我们在查字典的时候，如果没有目录，那我们就只能一页一页的去找我们需要查的那个字，速度很慢。如果有目录了，我们只需要先去目录里查找字的位置，然后直接翻到那一页就行了。

在 MySQL 中，无论是 Innodb 还是 MyISAM，都使用了 B+树作为索引结构。

索引的优缺点？

优点：

  1. 大大加快数据检索的效率，减少 IO 次数。
  2. 唯一索引可以保证数据的唯一性。

缺点：

  1. 索引会降低更新的效率。
  2. 索引会占用磁盘空间。

索引的分类有哪些？

从功能逻辑的角度，索引有普通索引、唯一索引、主键索引、全文索引。

从物理实现的角度，索引有聚簇索引和非聚簇索引。

从字段个数的角度，索引有单列索引和联合索引。

聚簇索引与非聚簇索引的区别？

  1. 聚簇索引叶子节点存储的是一条完整的记录，非聚簇索引叶子节点存储的是 主键+索引列 的组合
  2. 聚簇索引检索数据效率更快，非聚簇索引检索数据需要进行 回表查询，要检索两颗 B+ 树。

索引的底层数据结构？

B+ 树，所有数据存放在叶子节点上，数据构成一条单向链表。数据页构成双向链表。

B 树与 B+ 树的区别？

  • B 树的所有节点既存放键(key) 也存放数据(data)，而 B+树只有叶子节点存放 key 和 data，其他内节点只存放 key。
  • B 树的叶子节点都是独立的;B+树的叶子节点有一条引用链指向与它相邻的叶子节点。
  • B 树的检索的过程相当于对范围内的每个节点的关键字做二分查找，可能还没有到达叶子节点，检索就结束了。而 B+树的检索效率就很稳定了，任何查找都是从根节点到叶子节点的过程，叶子节点的顺序检索很明显。
  • 在 B 树中进行范围查询时，首先找到要查找的下限，然后对 B 树进行中序遍历，直到找到查找的上限；而 B+树的范围查询，只需要对链表进行遍历即可。

如何正确使用索引？索引失效的场景？

  • 创建了组合索引，但查询条件未遵守最左匹配原则;
  • 在索引列上进行计算、函数、类型转换等操作;
  • 以 % 开头的 LIKE 查询比如 LIKE '%abc';;
  • 查询条件中使用 OR，且 OR 的前后条件中有一个列没有索引，涉及的索引都不会被使用到;
  • IN 的取值范围较大时会导致索引失效，走全表扫描(NOT IN 和 IN 的失效场景相同);
  • 发生隐式转换

事务

什么是事务？

ACID

A：原子性，要么都成功，要么都失败。

C：一致性，执行事务前后，数据保持一致。

I：隔离性，并发访问时，一个用户的事务不被其他事务所干扰。

D：持久性，事务提交之后，它对数据库中数据的改变是持久的。

事务的隔离级别？

四种隔离级别

  • READ-UNCOMMITED（读未提交）：脏读、幻读、不可重复读都有可能发生。
  • READ-COMMITED（读已提交）：幻读和不可重复读可能会发生。
  • REPEATABLE-READ（可重复读）：幻读可能会发生
  • SERIALIZABLE(可串行化)：防止脏读、幻读、不可重复读

什么是脏读？幻读？不可重复读？

  • 脏读：一个事务读取到另一个事务还未提交的数据。
  • 幻读：在一个事务中对同一范围的数据进行两次查询，返回的结果集条数却不一样。
  • 不可重复读，在一个事务中对同一条数据两次查询，结果却不一样。

MySQL 有哪些事务日志？

  • redolog：重做日志，用于记录已提交事务对数据页的修改，在数据库发生故障时能恢复数据，保证了数据的持久性。
  • undolog：回滚日志，用于记录数据修改之前的数据，以便在事务回滚时可以恢复到事务开始的状态，保证了数据的原子性。

锁

什么是锁？

锁是用户管理多个事务并发访问数据的一种机制，事务的隔离性就是由锁来实现的。

MySQL 有哪些锁类型？

从锁的粒度来划分有表锁、页锁、行锁

表锁：

  • 共享锁和排他锁：也叫做读锁/写锁。
  • 意向锁：有意向共享锁和意向锁排他锁。
  • 自增锁：MySQL 自增时的锁。
  • 元数据锁：MySQL 修改表结构时的锁。

行锁：

  • 记录锁：也叫做行锁，锁定表中的一行记录。
  • 间隙锁：锁定的是一个范围，主要用于避免幻影记录的产生。
  • 临键锁：记录锁和间隙锁的组合，既锁定了一行记录，又锁定了该行之前的间隙。

表级锁是什么？行级锁是什么？他们的区别是什么？

  • 表级锁顾名思义就是对整个表进行加锁，锁粒度大，并发效率较低
  • 行级锁是在存储引擎层面实现的，InnoDB 支持行级锁，MyISAM 不支持；行级锁是针对索引字段加的锁，行级锁能对表中的某一条记录进行加锁，锁粒度较低，并发效率高，但会有死锁风险。

行锁什么情况下会导致死锁？如何避免？

并发事务的情况下，多个事务互相等待对方持有的锁，会导致死锁。例如：事务1 持有资源 A 并等待资源 B，而事务2 持有资源 B 并等待资源 A。

避免方法：

  1. 按相同的顺序访问资源，避免形成循环等待。
  2. 尽量减少事务的持有锁的时间，减少大事务。

意向锁的作用是什么？

意向锁主要是为了提高加锁的效率，在事务需要获取行锁或表锁之前，先获取对应的意向锁，这样可以快速判断是否有其他事务已经锁定了表或行，避免每行数据进行检查的开销。

MVCC

什么是 MVCC？

MVCC （Multiversion Concurrency Control），多版本并发控制。顾名思义，MVCC 是通过数据行的多个版本管理来实现数据库的 并发控制 。这项技术使得在InnoDB的事务隔离级别下执行 一致性读 操作有了保证。换言之，就是为了查询一些正在被另一个事务更新的行，并且可以看到它们被更新之前的值，这样在做查询的时候就不用等待另一个事务释放锁。

MVCC 解决了什么问题？

  • 提高并发性能：允许多个事务并发的读取和修改数据，而不会相互阻塞。
  • 解决读写冲突：读事务不会被写事务阻塞，写事务也不会阻塞读事务，提高了系统的吞吐量。
  • 实现事务的隔离性：不同的事务可以看到不同版本的数据，从而实现了事务的隔离级别。

MVCC 的工作流程？

  1. 首先获取事务自己的版本号，也就是事务 ID；

  2. 获取 ReadView；

  3. 查询得到的数据，然后与 ReadView 中的事务版本号进行比较；

  4. 如果不符合 ReadView 规则，就需要从 Undo Log 中获取历史快照；

  5. 最后返回符合规则的数据。

ReadView 的规则是什么？

ReadView 有几个关键的字段：

  1. creator_trx_id ，创建这个 Read View 的事务 ID。

    说明：只有在对表中的记录做改动时（执行INSERT、DELETE、UPDATE这些语句时）才会为事务分配事务id，否则在一个只读事务中的事务id值都默认为0。

  2. trx_ids ，表示在生成ReadView时当前系统中活跃的读写事务的 事务id列表 。

  3. up_limit_id ，活跃的事务中最小的事务 ID。

  4. low_limit_id ，表示生成ReadView时系统中应该分配给下一个事务的 id 值。low_limit_id 是系统最大的事务id值，这里要注意是系统中的事务id，需要区别于正在活跃的事务ID。

具体规则如下：

  • 如果被访问版本的trx_id属性值与ReadView中的 creator_trx_id 值相同，意味着当前事务在访问它自己修改过的记录，所以该版本可以被当前事务访问。

  • 如果被访问版本的trx_id属性值小于ReadView中的 up_limit_id 值，表明生成该版本的事务在当前事务生成ReadView前已经提交，所以该版本可以被当前事务访问。

  • 如果被访问版本的trx_id属性值大于或等于ReadView中的 low_limit_id 值，表明生成该版本的事务在当前事务生成ReadView后才开启，所以该版本不可以被当前事务访问。

  • 如果被访问版本的trx_id属性值在ReadView的 up_limit_id 和 low_limit_id 之间，那就需要判断一下trx_id属性值是不是在 trx_ids 列表中。

    • 如果在，说明创建ReadView时生成该版本的事务还是活跃的，该版本不可以被访问。
    • 如果不在，说明创建ReadView时生成该版本的事务已经被提交，该版本可以被访问。
